Session 2

GitHub as Your Knowledge Base

The conceptual shift. We stopped thinking of GitHub as "a developer thing" and started seeing it as the best place to store everything your AI needs to know about you. We also got our first look at the CLI and saw a real-world QA demo.

The Problem We Solved

The old way

You create a skill file. You want to share it with Banita. You download it, put it on a cloud drive, she downloads it, uploads it to her Claude UI. Next week you update the skill. Repeat the entire process. Every. Single. Time.

The GitHub way

You put your skill in a GitHub repo. Banita gets access. Her Claude reads directly from your repo. When you update the skill, Banita's Claude gets the update automatically. One update, everyone benefits.

Why GitHub Beats Folders

1

Version history. Every change is tracked. You can always go back. Unlike a folder where you overwrite and lose the previous version.

2

Live updates. Update once, everyone sees it. No download-upload cycle.

3

API access. Claude (and other AI tools) can read from GitHub directly via tokens. They can't read from your Google Drive or local folder.

4

Collaboration. Multiple people can contribute to the same repo. Pull requests ensure no one accidentally overwrites someone else's work.

5

Developers already use it. By storing your knowledge on GitHub, you're speaking the same language as the development team.

Personal vs Org Repos

Personal repo
Your own GitHub account. Private to you. For your CLAUDE.md, personal skills, preferences. No PR required — you commit directly. This is what you set up in Session 3.

Org repo
Dhwani RIS shared repository. Contains Security MD, domain skills, coding standards. Changes require a PR and developer approval. Coming soon — Ankit will share access.

CLI Introduction

We got our first look at Claude Code — the command-line interface. Key advantages over the chat UI:

+

No time limits. CLI continues indefinitely. No session timeouts.

+

Context preservation. If it stops, it picks up where it left off. Full history is saved.

+

Parallel agents. Give it 3 tasks and it sends 3 agents simultaneously. No waiting in line.

+

MCP connections. Connect to Gmail, Slack, Calendar, Apple Reminders — directly from the terminal.

+

Side questions. Ask a quick question without disrupting the main task using /btw.

CLI is not a replacement for the UI. Use UI for thinking, brainstorming, document creation, image generation, sharing conversations. Use CLI for coding, parallel tasks, integrations, and long-running work. Use both.

Real-World Demo: Dashboard QA via Video

Sunchay demonstrated a powerful technique for testing dashboards:

1

Record a 2-minute screen recording of the dashboard, applying filters and scrolling through data.

2

Compress to under 20 MB (Claude's attachment limit).

3

Feed to Claude with a skill that says: read every 2 frames per second, extract all data points, compare against:

A

The logic sheet (expected values)

B

The API response (actual data)

C

The UI screenshots (displayed values)

Result: Claude identified matching data, flagged mismatches, and even warned about KPIs that were technically correct but might not represent what the user intended. Three-layer validation from a 2-minute video.

Key Takeaways

1

GitHub is a knowledge repository, not just a code repository. Use it for skills, CLAUDE.md, domain knowledge — not just code.

2

Personal repos are your starting point. No PRs, no approval. Just you and your knowledge base.

3

CLI and UI serve different purposes. Neither replaces the other. Learn when to use which.

4

Video-based QA is real. You can test dashboards by recording, compressing, and feeding to AI.
