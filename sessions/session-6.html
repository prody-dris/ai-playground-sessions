The Craft — Clean Code, Good Design, Real Skills.

Session 6 · Thursday, 23 April 2026 · 5:00 – 6:00 PM
Presented by Abhijit Jamadagni & Aditya Gore (joint) · Abhinav · Hosted by Nihaan Mohammed

Why This Session

Session 5 left us with a coaching moment: Arpita’s Welspun Help Center was a great build — but it missed the mGrant design system skill, and the UI diverged from V2/V3. That was not an Arpita problem. That was a we-don’t-have-the-engineering-muscle-yet problem.

Session 6 is the answer. What does clean AI-assisted code actually look like? What does a skill need to contain for the next person not to repeat the mistake? How do we keep design consistency without a designer in every room? Three voices from the inside — Abhijit Jamadagni and Aditya Gore together on the coding craft, and Abhinav on the design craft — followed by a workshop where everyone writes a skill from a real correction they’ve made.

You’ll walk out with

  • A shared definition of “clean” AI-assisted code — review practices, hygiene rules, when to trust AI and when to push back
  • Battle-tested patterns from Abhijit & Aditya Gore’s last few months of daily use — what stuck, what failed, what they’d never do again
  • Design consistency as a skill, not a style guide — why putting the mGrant design system in your MD is how it scales
  • One real skill you wrote in the room, committed to dhwani-ris/skills — the first contribution to the organisation tier

Design system Skill-based Consistency

The Workshop 10 min, live

Nihaan asked the room at the end of S5: “Let’s build common skills in one common place — GitHub — so Deepak’s coding skill, Anup’s coding skill, are all available to the rest of us.” The workshop is the commit moment.

The exercise

Pick one correction you’ve made to AI twice this month. Not a one-off annoyance — a correction you’ve given AI more than once on the same kind of task. That’s your skill candidate.

Open the Skills Template. Fill the five blocks — name, description, context, procedure, example. Commit to dhwani-ris/skills with a PR. By 5:55 we’ll review the first batch live.

No perfection expected. First version of a skill is always wrong. The whole point of writing it is to find out how it’s wrong, on real work, over the next week. Ankit said it on the S5 call: “Skill banana is not one-shot. You iterate. You train it.”

Before You Come

Three small things so the hour lands hard. None of them will take more than ten minutes.

Pre-session homework

List two corrections. Think of two times in the last month you had to tell AI the same thing twice. What was the correction? What was the context? That’s the raw material for your skill.
Read your own CLAUDE.md. If you don’t have one, write a first draft — the CLAUDE.md template takes five minutes. Skills work best when they sit on top of a project that already knows who you are.
Bring your laptop with CLI installed. If Claude Code CLI isn’t installed yet, run the CLI setup prompt before Thursday. The workshop is CLI-first; we don’t want anyone stuck on install while the rest are writing.
