Four Concepts, One Operating System

These aren’t alternatives. They’re layers. Together they turn general-purpose AI into an org-aware, domain-specific, context-persistent system.

📄
Skill
A markdown file that teaches AI how your org works. Procedural knowledge, domain rules, workflows.
Procedural knowledge Persists across sessions Selective loading
🔒
MCP — Model Context Protocol
Open standard connector. Gives AI live access to your tools — email, calendar, database, HRMS, GitHub.
Live data access External systems Open standard
📦
Plugin
A distributable package bundling skills + MCP configs + hooks. Admin installs once — team gets the capability.
Bundled distribution Org-wide deployment Turnkey
🤖
Agent
A specialised AI worker with its own context window, tools, and task. Multiple agents run in parallel simultaneously.
Isolated context Parallel execution Delegation model

MCPs give AI access to your systems. Skills give AI your knowledge. Plugins distribute both across the team. Agents run them in parallel. They are complementary, not competing.

Agent Skills — The New Frontier Dec 2025

In December 2025, Anthropic released Agent Skills as an open standard. Partners — Atlassian, Figma, Canva, Stripe, Notion, Zapier — adopted immediately. The core innovation: skills that load only what’s needed, when it’s needed. Three tiers. Progressive disclosure.

Tier 1
Always Loaded
Metadata — Name & Description only Loaded at startup. Claude reads these to know what skills exist. No content, no procedures — just enough to decide relevance.
~100tokens/skill
Tier 2
On Relevance
Instructions — Workflows, domain rules, procedures Loads only when Claude determines the skill applies to the current task. An M&E skill doesn’t load when you’re writing code.
~5ktokens if triggered
Tier 3
On Demand
Resources — Templates, scripts, reference files Never loaded into context. Accessed on demand via file reads. Bundle 50+ reference documents with zero additional token cost.

70–90% reduction in baseline context consumption vs. static custom instructions. Admins provision skills org-wide. New hires arrive with full Dhwani RIS context from day one — zero individual setup.

How to Write and Maintain a Skill

01
Frontmatter — Name, description, trigger conditions. The description must be specific: “Use when creating MCA Section 135 CSR reports” beats “CSR skill.” This is the only part that always loads.
02
Context block — Why this skill exists and what domain knowledge underpins it. Written for AI to understand purpose, not just procedure.
03
Procedure — Step-by-step, with explicit decision points. Don’t leave judgment gaps. “If X, then Y. If Z, then ask.”
04
Constraints — What AI must never do in this skill’s context. Explicit limits beat hoping AI infers them. “Never send without human review.”
05
Example — At least one real example of the correct output. AI calibrates quality against examples, not abstract descriptions.

Evaluating a Skill

1

Define the baseline first. What should AI produce when this skill is active? Write it before writing the skill. This is your pass/fail.

2

Test on real work. Not a toy example. The M&E skill gets tested on an actual grant reporting scenario.

3

Gap = next iteration. Where AI deviated is where the skill needs more specificity.

4

Version and review quarterly. Stale skills are worse than no skills — they give AI confident, outdated instructions.

The rule: If you correct AI for the same mistake twice on the same task type, that correction belongs in a skill — not in your next prompt.

The Builder Protocol — Skills as SDLC

Skills are not standalone. The Builder Protocol is the system that sequences them into a repeatable build process. Three phases. Each has its skills. You don’t skip ahead.

Key Takeaways

1

Skills are the OS, not a feature. Without them, every session starts blank. With them, AI arrives knowing your rules.

2

MCPs give access. Skills give knowledge. You need both. They’re complementary.

3

Agent Skills changes token economics. 50+ skills available with ~5,000 tokens of overhead. The skill library can grow without the context window becoming a bottleneck.

4

Domain expertise belongs in skills, not prompts. If you re-explain the same context twice, write a skill.

5

A skill without an eval is a hope. Write the pass/fail before you write the skill. Test on real work. Update quarterly.

6

The Builder Protocol is Theme → Form → Dashboard. Not a framework to admire — a sequence to follow. Kashish has lived it.
